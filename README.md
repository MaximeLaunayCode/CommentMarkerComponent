# Marker Comment GitLab CI Component

`marker-comment` adds a canonical Marker Comment Block to supported files that are added by a merge request. It runs locally in the Git checkout, never calls the GitLab API, and emits an apply-ready `marker-comment.patch` instead of changing the source branch.

## Use the component

Production pipelines should pin an exact semantic release. The `workflow` rules must be defined directly in the consuming project: rules inside an included component cannot cause GitLab to create merge request pipelines.

```yaml
stages:
  - test

workflow:
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH'

include:
  - component: $CI_SERVER_FQDN/platform/ci-components/marker-comment@1.0.0
    inputs:
      stage: test
      marker-text: "Generated by internal platform policy"
```

The component contributes exactly one merge-request-only job. Do not pin production use to a branch or `~latest`; choose a reviewed semantic version such as `1.0.0`.

## Inputs

| Input | Type | Default | Meaning |
| --- | --- | --- | --- |
| `marker-text` | string | required | Managed Marker Body; must contain non-whitespace text. |
| `job-name` | string | `marker-comment` | Name of the job added to the consumer pipeline. |
| `stage` | string | `test` | An existing stage in the consumer pipeline. |
| `fail-when-patch-needed` | boolean | `true` | Exit `1` when a patch is needed. `false` enables Report-Only Mode. |
| `file-globs` | array | `[]` | Patterns that narrow the supported-file allowlist. |
| `exclude-globs` | array | `[]` | Extra exclusion patterns. |
| `begin-sentinel` | string | `MARKER-COMMENT: BEGIN` | Exact begin label for the managed block. |
| `end-sentinel` | string | `MARKER-COMMENT: END` | Exact end label for the managed block. |

Sentinels must be distinct, non-empty single lines without surrounding whitespace or NUL bytes. Marker text uses universal line endings; trailing whitespace is removed per logical line while leading whitespace and interior blank lines are preserved.

## Supported syntax

Matching is case-sensitive. `file-globs` can only narrow these mappings; it cannot opt an unsupported syntax in.

| Syntax | Files |
| --- | --- |
| Hash line (`#`) | `.sh`, `.bash`, `.zsh`, `.ksh`, `.fish`, `.py`, `.rb`, `.yml`, `.yaml`, `.toml`; `Dockerfile`, `Dockerfile.*`, `Makefile`, `GNUmakefile`, `*.mk` |
| Slash line (`//`) | `.js`, `.jsx`, `.mjs`, `.cjs`, `.ts`, `.tsx`, `.java`, `.c`, `.h`, `.cc`, `.cpp`, `.cxx`, `.hpp`, `.cs`, `.go`, `.kt`, `.kts`, `.swift`, `.rs` |
| Bang line (`!`) | `.f90`, `.f95`, `.f03`, `.f08`, `.f18` |
| HTML block (`<!-- -->`) | `.html`, `.htm`, `.xml`, `.md`, `.markdown`, `.vue`, `.svelte` |
| CSS block (`/* */`) | `.css`, `.scss`, `.sass`, `.less` |

Legacy fixed-form Fortran (`.f`, `.for`, `.ftn`) is not supported. CSS content containing `*/` and HTML content containing `--` cannot be rendered safely; the affected file is reported and left unchanged.

## Glob behavior

Globs use case-sensitive Git-wildmatch-style semantics against repository-relative paths:

- `/` separates segments; `*` and `?` do not cross `/`, while `**` can.
- Character classes are supported.
- A pattern without `/` matches a basename at any depth.
- A leading `/` anchors the match at the repository root.
- Negation and directory-only patterns are not supported.

Every pattern is validated before processing. A path must first be an MR-Added File with supported syntax, then match at least one `file-globs` pattern when configured, and finally survive built-in and consumer exclusions. Exclusion always wins.

### Built-in exclusions

The component excludes:

- any dot-prefixed path segment, including dotfiles;
- lockfiles named `package-lock.json`, `npm-shrinkwrap.json`, `yarn.lock`, `pnpm-lock.yaml`, `bun.lock`, `bun.lockb`, `Cargo.lock`, `Gemfile.lock`, `poetry.lock`, `uv.lock`, `Pipfile.lock`, `composer.lock`, `go.sum`, `gradle.lockfile`, `packages.lock.json`, `Podfile.lock`, `mix.lock`, `pubspec.lock`, or `flake.lock`;
- path segments named `vendor`, `vendors`, `third_party`, `third-party`, `node_modules`, or `bower_components`;
- NUL-containing files, symlinks, Git submodules, and non-regular Git entries.

There is no size cutoff or generated-file inference in v1.

## History and merge request requirements

The job requires a real merge request pipeline and GitLab's merge request variables. It sets `GIT_DEPTH: "0"`, fetches the target ref without tags, and verifies the diff base locally. The runner must therefore have full, fetchable history for the target project. In v1, fork merge requests are unsupported when their pipeline cannot fetch the parent project's target history; the job exits with discovery guidance.

No API token is accepted or required. Discovery uses only local Git. The released job does not commit or push, and it does not need credentials for the merge request API.

## Patch artifact and developer workflow

When changes are needed, the job creates the fixed path `marker-comment.patch`. GitLab collects it with `artifacts: when: always`, names the artifact `marker-comment-$CI_COMMIT_SHORT_SHA`, and retains it for one week. No empty patch is created, so clean runs have no patch artifact. A partial patch remains collectable when another file has a processing error.

Download the artifact at the clean source head, then run:

```sh
git apply --check marker-comment.patch
git apply marker-comment.patch
git add -A
git commit -m "Apply marker comment policy"
git push
```

Review the patch before committing. The component itself does not commit or push.

### Exit codes

| Code | Meaning |
| --- | --- |
| `0` | No patch is needed, or Report-Only Mode found a needed patch. |
| `1` | A patch is needed and `fail-when-patch-needed` is `true`. |
| `2` | Configuration, discovery, malformed-marker, or processing error; a partial patch can still exist. |

## V1 boundaries

- Merge request pipelines only; no push-pipeline mode.
- No automatic commit or push and no API fallback.
- No configurable patch path, unsupported-syntax override, sentinel migration, generated-file inference, or file-size cutoff.
- Consumer edits inside a Managed Marker Body are overwritten with the configured canonical text.

## Release process

Push a semantic-version tag such as `1.0.0`. The tag pipeline runs component compilation, all processor and consuming-project fixture tests, builds and publishes `marker-comment:$CI_COMMIT_TAG`, proves that the sample patch applies and reruns cleanly, and only then creates the GitLab release that publishes the component. The component's `$[[ component.version ]]` image reference therefore resolves to the same immutable semantic version.
